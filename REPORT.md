## 1. Introduction

### 1.1 Motivation

Scaling laws have become a foundational tool in modern deep learning. Kaplan et al. (2020) showed that for natural language models, validation loss decreases as a power law in the number of model parameters, training tokens, and compute: $L = a \cdot N^{-\alpha} + c$. These predictable relationships have shaped how labs allocate compute and design model families.

A natural question is whether such scaling laws are universal, or whether they depend on properties of the data domain. Most scaling-law work has focused on natural language. In this project we examine a structured, non-linguistic domain: **Scalable Vector Graphics (SVG) code**. SVG is a useful test case for several reasons:

- It is XML-based text with strict syntactic rules — invalid output is detectable mechanically.
- It has a hierarchical structure (nested groups, transforms) that resembles code more than prose.
- Its outputs render to images, so generation quality can be assessed both quantitatively (validity, render rate) and qualitatively (visual coherence).
- The vocabulary is more constrained than natural language — a small number of tag names, attribute names, and numeric tokens dominate.

A second open question concerns hyperparameter tuning at scale. Under standard parameterization (SP), the optimal learning rate for a small "proxy" model does not transfer to wider models. **Maximal Update Parameterization (µP)**, introduced by Yang et al. (2022), reparameterizes neural network layers so the optimal learning rate is invariant to width. We test whether µP delivers measurable improvements in scaling behavior on SVG data — a setting outside the natural-language regime where µP was originally validated.

### 1.2 Contributions

This project makes the following contributions:

1. A complete data-preprocessing pipeline for SVG code, including normalization, deduplication, validation, and BPE tokenization.
2. A scaling study of decoder-only transformers from ~1M to ~88M parameters trained on SVG, with a fitted power-law exponent.
3. A comparison of standard parameterization against µP, including a learning-rate sweep under each parameterization.
4. An extrapolation of the fitted scaling law to predict loss for a 10× larger model.
5. Quantitative and qualitative evaluation of generated SVG samples, including unconditional and prefix-conditioned generation.

## 2. Data

### 2.1 Datasets

The primary dataset is **`starvector/svg-icons-simple`** (Rodriguez et al., 2023), a collection of approximately 89,370 simplified SVG icons curated for the StarVector project. The dataset is "simplified" in the sense that complex SVG features (gradients, filters, animations, embedded raster images) have been removed by the StarVector team, leaving clean SVGs that consist primarily of `<path>`, `<circle>`, `<rect>`, `<polygon>`, and `<g>` elements with basic styling.

To reach the 100M training-token target, we supplemented with two additional StarVector datasets: **`starvector/svg-emoji-simple`** (4,114 simplified emoji SVGs) and a 65% subsample of **`starvector/svg-fonts-simple`** (1,134,108 of 1,744,783 font glyph SVGs). The fonts dataset provides simple, single-path glyphs that are structurally similar to icons but with greater diversity of path shapes. In total, we loaded 1,218,656 raw SVGs across the three sources.

### 2.2 Cleaning and Normalization Pipeline

Each raw SVG is processed through the following pipeline, in order:

1. **Strip comments and metadata.** XML comments (`<!-- ... -->`), processing instructions (`<?xml ?>`), and `<metadata>`, `<desc>`, `<title>` blocks are removed. These contain authoring tool information and human-readable annotations that do not affect rendering.
2. **Extract the `<svg>...</svg>` root.** Any text outside the SVG element is discarded.
3. **Round numeric values to one decimal place.** All decimal numbers in attributes and path data are rounded. For example, `M10.059374809265137 7.275000095367432` becomes `M10.1 7.3`. Integers are kept clean (e.g., `0.0` becomes `0`).
4. **Collapse whitespace.** Multiple spaces and blank lines are collapsed to single spaces and newlines.
5. **Validate XML.** Each cleaned SVG is parsed with `lxml.etree.fromstring`. Invalid XML is discarded.
6. **Length filter.** SVGs shorter than 50 characters are discarded as trivial.
7. **Deduplicate.** Cleaned SVGs are hashed (MD5) and duplicates are removed.

#### Design decisions

| Decision | Choice | Justification |
|---|---|---|
| Coordinate precision | 1 decimal place | At a 24×24 viewBox typical for icons, 0.1 units is sub-pixel at standard rendering resolutions. The visual impact is imperceptible but the vocabulary impact is large: the tokenizer no longer needs to learn thousands of unique 15-digit float strings. |
| Min length | 50 characters | Removes empty wrappers and trivial SVGs that contain no meaningful structure. |
| Deduplication | MD5 of cleaned SVG | Prevents the model from memorizing duplicated icons and inflating perplexity metrics. |
| XML validation | Strict `lxml` parsing | Guarantees that 100% of training data is syntactically valid, so the model never learns from broken examples. |

#### Cleaning results

Across all three datasets, the pipeline processed 1,218,656 SVGs and retained 1,170,625 (96.1%). The 48,031 removals were all duplicates — zero SVGs were rejected for invalid XML, missing `<svg>` root elements, or being too short. This reflects the quality of the StarVector pre-processing: the raw data is already well-formed. The emoji dataset had the highest duplicate rate (19.6% in a sample of 500), while icons and fonts had effectively none.

Although the pipeline removed few files, the normalization step produced substantial compression. For a representative icon SVG:

- Raw length: 2,235 characters
- Cleaned length: 919 characters
- **Reduction: 59%**

Coordinate rounding accounts for most of this compression. For example: `viewBox="0.0 0.0 24.0 24.0"` → `viewBox="0 0 24 24"`, and `M10.059374809265137 7.275000095367432` → `M10.1 7.3`.

This compression is important because it reduces the number of unique numeric tokens BPE must learn, which makes the vocabulary more efficient.

### 2.3 Tokenization

We trained a Byte Pair Encoding (BPE) tokenizer on the cleaned corpus using the HuggingFace `tokenizers` library, with a byte-level pre-tokenizer for full Unicode coverage.

**Vocabulary size: 4,096.** This was chosen as a compromise between two concerns:

- Smaller vocabularies (e.g., 1,024) produce longer sequences. With more tokens per SVG, the model spends more capacity learning basic syntax.
- Larger vocabularies (e.g., 8,192+) introduce many rare tokens whose embeddings receive too few gradient updates to be learned well, given our ~100M training-token budget.

At 4,096 tokens with ~130M training tokens, each token receives an average of ~32,000 occurrences — sufficient to learn meaningful embeddings.

**Special tokens:** `<PAD>` (id 0), `<BOS>` (id 1), `<EOS>` (id 2), `<UNK>` (id 3).

The tokenizer was trained on a random sample of 150,000 SVGs from the cleaned corpus (training on the full 1.17M was unnecessary and exceeded Colab memory limits during the BPE merge computation).

**Tokenization behavior.** The BPE tokenizer learned highly efficient SVG-specific tokens. Notably, the entire SVG header and common path preamble — `<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 24 24" height="200px" width="200px"><path fill="none" stroke="black" stroke-width=".3" stroke-opacity="1" filling="0" d="M...` — was captured as a single token. Coordinate values like `7.3` and path commands like `C14.2` also became individual tokens. This compression reduces a typical 919-character icon SVG to just 96 tokens.

The most frequent non-special tokens were coordinate values in the 0–24 range (matching the standard 24×24 viewBox), confirming that the tokenizer adapted well to SVG-specific patterns.

### 2.4 Splits and Final Statistics

We split the cleaned corpus into 98% train, 1% validation, 1% test **by file** (not by token position within the concatenated corpus). Splitting by file prevents data leakage — without it, the beginning of an SVG could appear in training and the end in validation.

| Split | # Files | # Tokens |
|---|---|---|
| Train | 1,147,128 | 130,404,653 |
| Val | 11,705 | 1,326,073 |
| Test | 11,705 | 1,341,361 |
| **Total** | **1,170,538** | **133,072,087** |

The training set contains 130.4M tokens, exceeding the 100M minimum. Sequence lengths have a median of 98 tokens (P95: 246, P99: 447), with only 65 SVGs exceeding the 1,024-token maximum and being filtered out.
